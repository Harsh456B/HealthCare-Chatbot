## What This Project Does

```
User Question (Chat Interface)
    ↓
[Retrieval] Search vector database for relevant documents
    ↓
[Augmentation] Combine documents with prompt template
    ↓
[Generation] LLM generates response using context
    ↓
Response (Limited to 3 sentences for safety)
```

This is a **Retrieval-Augmented Generation** (RAG) system, not a simple Q&A bot or fine-tuned model.

## Key Learnings from This Project

### Architecture
- **Separation of Concerns**: Backend logic separate from routes
- **Pipeline Pattern**: Document processing as clear stages
- **Statelessness**: Each request independent (scales horizontally)

### Technology Choices
- **Why Groq?** - Fast inference on large models
- **Why Pinecone?** - Managed vector DB (no ops burden)
- **Why HuggingFace embeddings?** - Good quality-speed tradeoff
- **Why Flask?** - Lightweight Web framework perfect for APIs
- **Why RAG over fine-tuning?** - Faster, cheaper, easier updates

### Best Practices Demonstrated
- Environment variables for secrets ✅
- Type hints throughout codebase ✅
- Docstrings on functions ✅
- Modular file organization ✅
- Clear variable naming ✅
- Single responsibility functions ✅
- Error handling with defaults ✅
- Input HTML escaping ✅

### Common Mistakes AVOIDED
- Hardcoded API keys ✅
- Mixing business logic with routes ✅
- Weak error messages ✅
- XSS vulnerabilities ✅
- No separation of concerns ✅
- Monolithic files ✅
